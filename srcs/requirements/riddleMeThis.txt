1. PID 1 - alle services der Docker container laufen auf PID1 - Parent process
    -   falls etwas schief gehen sollte, stürtzt der Docker container aber
#   - container lebt nur solange wie PID1
#   - Docker sendet Stop SIgnale an PID1
#   - PID sollte sauber Prozesse beenden können
#   - PID sollte idealerweise Zombie Prozesse einsammeln
Warum Service als PID1?
#   - wenn Hauptservice crashed, endet der Container sichtbar\
#   - Orchestration/Restart-Policy kann sauber reagieren
#   - Logs & Fehler kann man klarem Prozess zuordnen
#   - man kann auch Services als Child process haben 
#       aber dann braucht man Supervisor (Signal-Forwarding, Reaping, ...)
