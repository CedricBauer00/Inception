Inception - Docker Network Infrastructure

This project has been created as part of the 42 curriculum by cbauer

Description

This project named inception is about deepening your knowledge about system administration by using Docker. You will create a small infrastructure containing different services with specific rules, given by the subject, by writing Dockerfiles and orchestrating them using Docker Compose. The different services you are using are NGINX, WordPress and MariaDB, which will all be implemented in separate Docker containers. The basic goal is to create a personal web server setup with those three services, which will be connected via a custom Docker network.

Instructions

Prerequisites

  • Docker Engine
  • Docker Compose
  • Make

Installation & Execution

  1. Clone the repository: 
    git clone "repository URL"
cd Inception
  2. Setup environment variables:
    • Ensure .env is present in srcs folder
    • Ensure that the secrets folder is not containing any credentials
  3. Build and launch the infrastructure: 
    make
    By running make, all of the Docker images will be built and the containers will be started by the docker-compose.yml
  4. Domain Setup:
    • Add the following line to your /etc/hosts file and replace login with your username:
    127.0.0.1 login.42.fr
  5. Access the network:
    • Open the browser and enter the domain, so you will be connected to https://login.42.fr

Project Description

Docker & Sources

We are using Docker to containerize the services. As previously mentioned, the services run in isolated containers which are orchestrated and managed by the docker-compose.yml file. The docker-compose coordinates the network. The sources are located in the srcs/requirements folder.

Design

  • Operating System: The host machine runs Debian 12.12.
  • Base Images: The containers (NGINX, MariaDB, WordPress) are built from Debian 12 (Bookworm) for stability.
  • Security (TLS): NGINX is used as a reverse proxy and is the only entry point to the network, via port 443, using TLSv1.2/1.3.
  • Persistence: Data is stored in Docker volumes (wp_data, db_data), mounted at /home/user/data on the host. Bind mounts are strictly forbidden by the subject.

Technical Comparisons

Virtual Machines vs Docker

Feature VM Docker
Architecture Runs a full Guest OS on a Hypervisor Shares the host OS kernel - isolates processes in user space
Performance High overhead (CPU/RAM) for OS operations Lightweight - near-native performance
Isolation Hardware-level isolation Process-level isolation (namespaces & cgroups)
Size Large (Gigabytes) Small (Megabytes)

Secrets vs Environment Variables

  • Environment Variables
    • Storage: Plain text in configuration or memory
    • Visibility: Visible in docker inspect and env inside the container
    • Use Case: Internal configuration (paths, username, ...)
  • Docker Secrets
    • Storage: Encrypted on disk (Swarm) or managed securely
    • Visibility: Only mounted as files (e.g. /run/secrets/my_secrets) into the container
    • Use Case: Sensitive data (passwords, certificates)

Docker Network vs Host Network

  • Host Network
    • Isolation: Container shares the host's networking namespace
    • Port Mapping: No mapping needed - binds directly to host interface
    • Security: Low - directly exposed to the outside world
  • Docker Network
    • Isolation: Container gets its own IP and network namespace
    • Port Mapping: Ports must be explicitly published (-p 443:443)
    • Security: High - communication only allowed within the defined network; only one entrypoint

Docker Volumes vs Bind Mounts

  • Docker Volumes
    • Management: Managed by Docker (/var/lib/docker/volumes)
    • Portability: High - independent of host directory structure
    • Subject Rule: Mandatory for persistent database and web files
  • Bind Mounts
    • Management: Managed by user on host file system
    • Portability: Low - relies on specific paths existing on the host
    • Subject Rule: Forbidden for the main data storage in this project
